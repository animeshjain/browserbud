# Client-Side Transcript Extraction

How the `yt-research` skill extracts YouTube transcripts directly from the browser extension, avoiding paid API calls.

## Why client-side?

YouTube's captions API requires a Proof of Origin Token (POT) that can only be obtained by YouTube's own client-side JavaScript. External APIs (Supadata, ScrapeCreators) work around this at a cost. Since our extension already runs on youtube.com, we can piggyback on the player's own authenticated requests for free.

## How it works

The MAIN world script (`youtube-player.ts`) monkey-patches `XMLHttpRequest` at page load. When a transcript is requested, it toggles the CC button to force YouTube's player to fire an XHR to `/api/timedtext` — this XHR includes the POT token automatically. The interceptor captures the JSON3 response, parses it into timestamped segments, and sends it back through the message chain.

## Request flow

```
yt-research CLI
  │  fetchFromClient(videoId)
  │  POST /api/extract-transcript { videoId }
  ▼
server.js
  │  Holds HTTP response open
  │  Sends { type: "extract-transcript", videoId, requestId } via /ws/extension
  ▼
background.ts
  │  Finds YouTube tab where URL contains v={videoId}
  │  browser.tabs.sendMessage(tabId, { type: "extractTranscript", videoId, requestId })
  ▼
content.ts (ISOLATED world)
  │  window.postMessage({ type: "BROWSERBUD_EXTRACT_TRANSCRIPT", videoId, requestId })
  ▼
youtube-player.ts (MAIN world)
  │  Sets pendingExtraction so XHR interceptor knows to capture
  │  Loads captions module, toggles CC button
  │  YouTube's player fires XHR to /api/timedtext with POT token
  │  Interceptor captures JSON3 response → parses segments → formats text
  │  Restores CC button state
  │  window.postMessage({ type: "BROWSERBUD_EXTRACT_TRANSCRIPT_RESULT", ... })
  ▼
content.ts → browser.runtime.sendMessage({ type: "transcriptResult", ... })
  ▼
background.ts → extensionSocket.send({ type: "extract-transcript-result", ... })
  ▼
server.js → resolves pending request → caches transcript → HTTP 200 to CLI
  ▼
yt-research CLI receives { ok: true, text, lang, meta, source: "client" }
```

## Components involved

| File | Role |
|------|------|
| `skills/yt-research/transcript.ts` | `fetchFromClient()` — first provider in the chain, POSTs to server |
| `sprite/server.js` | `POST /api/extract-transcript` endpoint + `/ws/extension` WebSocket server |
| `extension/entrypoints/background.ts` | WebSocket client (auto-reconnects), dispatches commands to content scripts |
| `extension/entrypoints/content.ts` | Bridges messages between background (runtime.sendMessage) and MAIN world (postMessage) |
| `extension/entrypoints/youtube-player.ts` | XHR interception, CC toggle, JSON3 parsing |

## Bidirectional WebSocket

Before this feature, communication was one-way: extension → server. The server had no way to ask the extension to do something.

The `/ws/extension` WebSocket (on port 8989, same HTTP server as the proxy) adds a reverse channel:

- **background.ts** connects on startup, reconnects every 3s on disconnect
- **server.js** tracks the single connected extension WebSocket and a `pendingRequests` map (requestId → {resolve, reject, timer})
- When the CLI hits `/api/extract-transcript`, the server generates a `requestId`, sends the command on the WebSocket, and holds the HTTP response until the result comes back (or 15s timeout)

## Timeouts

Layered so each level times out before its parent:

| Layer | Timeout | On timeout |
|-------|---------|------------|
| youtube-player.ts (XHR interception) | 10s | Sends error result via postMessage, restores CC state |
| server.js (pending request) | 15s | Rejects with 502, cleans up pendingRequests map |
| transcript.ts (HTTP fetch) | 20s | Catches error, falls through to Supadata |

## Provider chain

`fetchTranscript()` in `transcript.ts` tries providers in order. The first success wins; failures log and continue:

1. **client** — `fetchFromClient()` → hits `/api/extract-transcript` (free, requires user on video page)
2. **supadata** — server-side Supadata API (paid)
3. **scrapecreators** — server-side ScrapeCreators API (paid, if key is configured)

## When client extraction fails

The client provider silently falls through to server-side APIs when:

- Extension WebSocket is not connected → server rejects immediately
- User is not on the requested video → background can't find a matching tab
- Video has no CC button → youtube-player.ts returns error
- XHR interception doesn't fire within 10s → timeout in youtube-player.ts
- YouTube changes timedtext XHR format → parsing fails

## The POT token problem

YouTube's `/api/timedtext` endpoint returns `content-length: 0` without a valid `pot` (Proof of Origin Token) parameter. This token:

- Is generated by YouTube's client-side JS (not available in page HTML or any config object)
- Is appended to the timedtext URL by the player's internal XHR, not by any public API
- Cannot be constructed or extracted — only captured by intercepting the player's own request

This is why the XHR interception approach works but direct fetch/API calls don't. The Innertube Player API (`/youtubei/v1/player`) also returns `UNPLAYABLE` for many videos now, making Strategy 1 from the original research doc unreliable.

## Testing

```bash
# Check extension WebSocket is connected (look for [ext-ws] in server logs)
node sprite/server.js

# Test extraction directly (user must be on a YouTube video page)
curl -X POST http://localhost:8989/api/extract-transcript \
  -H "Content-Type: application/json" \
  -d '{"videoId":"BzKubuqm1lQ"}'

# Test via the skill CLI
npm run --prefix skills/yt-research cli -- transcript BzKubuqm1lQ
```

Server log tags:
- `[ext-ws]` — extension WebSocket connect/disconnect
- `[extract]` — transcript extraction requests and results

## JSON3 format

YouTube returns captions in JSON3 format (requested via `&fmt=json3`). Structure:

```json
{
  "wireMagic": "pb3",
  "events": [
    {
      "tStartMs": 480,
      "dDurationMs": 4080,
      "segs": [
        { "utf8": "hello " },
        { "utf8": "everyone ", "tOffsetMs": 400 }
      ]
    }
  ]
}
```

Parsed into: `[{ startMs: 480, text: "hello everyone" }, ...]`

Formatted as: `[00:00] hello everyone`
